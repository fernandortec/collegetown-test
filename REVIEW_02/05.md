# Code Review — Better VPing (Phase 4 codebase)

> Reviewed: full `apps/api` and `apps/web` trees as of 2026-05-03  
> Reviewer note: structured as a PR review — **🔴 Blocking**, **🟡 Non-blocking / should fix**, **🔵 Suggestion / polish**

---

## Summary

The project is in a solid shape for a Phase 4 codebase. The scraper architecture is well-designed, the error taxonomy (`StaffExtractionError`) is clean, and the React / React Query integration is idiomatic. That said, there are a few blocking issues that will break production, a handful of correctness bugs, and several quality items worth addressing before Phases 5–9 build on top of this layer.

---

## 🔴 Blocking

### 1. CORS origin is hardcoded to `localhost` — production will break

**File:** `apps/api/src/app.ts:8`

```ts
const FRONTEND_ORIGINS = ["http://localhost:5173"];
```

This list is never populated from the environment. When the frontend is deployed to Vercel and the API to Render, every browser request will be blocked by CORS. The origin must come from an env var (`CORS_ORIGIN` or similar), with `localhost:5173` only as the local dev default.

---

### 2. Playwright image version is behind the installed package

**File:** `apps/api/Dockerfile:1`

```dockerfile
FROM mcr.microsoft.com/playwright:v1.56.1-noble AS build
```

`package.json` pins `playwright: ^1.59.1` (npm) but the Docker image is `v1.56.1`. The bundled browser binaries in the image will not match the npm-installed version. At minimum this produces a Playwright warning; in practice it can cause browser launch failures in production. Align both to the same patch version and pin the npm package exactly (`"playwright": "1.59.1"`) so they don't diverge again.

---

### 3. Internal scraper config is serialized into the API response

**File:** `apps/api/src/modules/school/school.types.ts`, `apps/api/src/modules/school/school.routes.ts:28`

The `DiffReport` type embeds a full `School` object, which includes:

```ts
scrapers: {
  current: StaffScraperConfig;
  archive: StaffScraperConfig;
};
```

`StaffScraperConfig` contains a `readySelector` string **and a Playwright `scrape` function**. `JSON.stringify` silently drops function-valued keys, so the response will serialize `scrapers: { current: { readySelector: "..." }, archive: { readySelector: "..." } }` — leaking internal DOM selectors in every `/diff` response body. The fix is to strip `scrapers` (and any other backend-only fields) from the JSON before returning, or introduce a dedicated response DTO that excludes them.

## 🟡 Non-blocking / Should Fix

### 4. `waitForSelector` failure is silently swallowed

**File:** `apps/api/src/modules/school/school.extraction.ts:80–91`

```ts
await page
  .waitForSelector(config.readySelector, { timeout: pageTimeoutMs })
  .catch(() => undefined);
```

If the CSS selector never appears (e.g., the site changed its markup), scraping continues on a page that hasn't fully loaded. The scraper then returns whatever partial DOM it can find, which passes the non-zero count guard only by luck. Consider distinguishing between "network idle timeout" (soft, OK to continue) and "content selector never appeared" (hard, should fail loudly) to surface real regressions.

---

### 5. Double deduplication between `normalizeStaffRecords` and `sanitizeStaffRecords`

**Files:** `apps/api/src/modules/school/scrapers/scraper-utils.ts:15–48`, `apps/api/src/modules/school/school.extraction.ts:187–224`

`normalizeStaffRecords` builds a `seen` set and filters duplicates. `sanitizeStaffRecords` (called immediately after in every code path) also builds a `seen` set and filters duplicates using the same key. One of these is redundant. Pick one place to own deduplication and remove the other; the current duplication is confusing to readers and wastes allocations on large staff lists.

---

### 6. Unused AI SDK dependencies

**File:** `apps/api/package.json:15–16`

```json
"@ai-sdk/google": "^3.0.67",
"ai": "^6.0.174"
```

Neither package is imported anywhere in the current source tree. The plan references Gemini Flash for Phase 3, but the implementation was replaced by direct DOM scraping (the correct and working approach). These packages add ~40 MB of production image weight for zero benefit. Remove them, and also remove the now-optional `GOOGLE_AI_API_KEY` from `env.ts` until Phase 8 actually needs it.

---

### 7. `ErrorResponse` type is defined twice

**Files:** `apps/api/src/@types/types.d.ts`, `apps/api/src/http/responses.ts:4–10`

The `ErrorResponse` type is defined identically in both files. The one in `@types/types.d.ts` appears to be a leftover from an earlier iteration. Delete it and import from `responses.ts` if needed elsewhere.

---

### 8. `SchoolPage` calls `useSchoolsQuery()` to find an already-cached school, but mishandles the `schoolsQuery.isError` path

**File:** `apps/web/src/pages/SchoolPage.tsx:18–29`

When `schoolsQuery.isError`, the page renders `<CatalogErrorPage>`. But `useSchoolDiffQuery` has already been called above with `enabled: schoolsQuery.isSuccess && Boolean(school)`. This is harmless today because `enabled: false` prevents the diff query from running. However, `diffQuery` is now an unresolved promise reference that hangs in the component — and the early return for `schoolsQuery.isPending` is placed *after* both query calls. If the hook call order ever changes (e.g., diffQuery is moved inside an `if`), React's rules of hooks will break. Keep query calls grouped at the top and gate rendering in the JSX only.

---

### 9. `SourceCard` and `StaffList` use `<Link>` for external URLs

**Files:** `apps/web/src/features/schools/components/SourceCard.tsx:23–29`, `apps/web/src/pages/SchoolPage.tsx:213–220`

```tsx
<Link to={url} rel="noreferrer" target="_blank">
```

`<Link>` is react-router's internal navigation component and is not meant for external `http://` URLs. While it works in practice because react-router falls back to a full navigation when given an absolute URL, the semantics are wrong and it could behave unexpectedly with future router versions. Use a plain `<a href={url}>` for all external links.

---

### 10. React key uses array index

**File:** `apps/web/src/pages/SchoolPage.tsx:224`

```tsx
key={`${record.name}-${index}`}
```

Mixing `record.name` with `index` doesn't make the key stable — if two records share a name or the array order changes, React will incorrectly reuse DOM nodes. Use only `record.name` if it is unique within the list, or `${record.name}|${record.title}` which matches the dedup key already used in the scraper layer.

---

### 11. `CatalogLoadingPage` hardcodes school names

**File:** `apps/web/src/shared/components/CatalogLoadingPage.tsx:7`

```tsx
{["Georgia", "Virginia Tech", "Wittenberg"].map((name) => (
```

This duplicates knowledge that lives in the catalog. If a school is added or removed, the loading skeleton will be wrong. Accept an optional count prop (defaulting to 3), or source the list from the same data the server returns.

### 12. `buildDiffReport` is misnamed for what it currently returns

**File:** `apps/api/src/modules/school/school.service.ts:26`

The function name and the `DiffReport` type name both imply a computed diff — added, removed, changed staff. Today the function just fetches two staff lists and returns them side by side. Until Phase 5 adds the actual diff engine, the function would be clearer as `buildExtractionReport` or `buildStaffComparisonReport`. The misleading name also makes it harder to spot in Phase 5 where the real diff logic should be inserted.

## 🔵 Suggestion / Polish

### 13. `import type { Page }` is out of place in `school.types.ts`

**File:** `apps/api/src/modules/school/school.types.ts:25`

```ts
export type StaffScraperConfig = {
  readySelector: string;
  scrape: (page: Page) => Promise<StaffRecord[]> | StaffRecord[];
};

import type { Page } from "playwright";   // ← out of order
```

The import sits in the middle of type definitions rather than at the top of the file. Move it to the top per TypeScript convention.

---

### 14. `extractGenericStaffRecords` is too long (~160 lines)

**File:** `apps/api/src/modules/school/scrapers/scraper-utils.ts:87–246`

The function mixes three distinct strategies (table row matching, card matching, and text sliding-window matching) in a single body. This makes it hard to test each strategy in isolation and hard to extend. Consider extracting `extractFromTableRows`, `extractFromCards`, and `extractFromText` helpers, then having `extractGenericStaffRecords` call them in order and merge results.

---

### 15. `withAlpha` has no validation

**File:** `apps/web/src/features/schools/utils.ts:10–12`

```ts
export function withAlpha(hex: string, alpha: string): string {
  return `${hex}${alpha}`;
}
```

This is just string concatenation. If `hex` arrives without the `#` prefix, or `alpha` is not a 2-digit hex string, the CSS color will be silently invalid and the school accent colors will disappear. Add a runtime guard or JSDoc clarifying the expected format.

---

### 16. `cleanupPageChrome` removes `<style>` tags

**File:** `apps/api/src/modules/school/school.extraction.ts:144`

Removing `style` elements from the live DOM is fine when using Playwright's `page.evaluate` for text extraction — but the same function is called before `waitForSelector`, which runs in the actual browser context. Stripping styles may cause dynamically styled elements (e.g., those hidden via `display:none` in CSS) to become visible, accidentally contributing junk records. A lighter approach: strip styles *after* waiting for the content selector, or document that the ordering is intentional.

---

### 17. `getJson` silently returns `null` on JSON parse failure

**File:** `apps/web/src/services/api.ts:36`

```ts
const jsonData: unknown = await response.json().catch(() => null);
```

If the response body is not valid JSON, `jsonData` is `null`. A successful HTTP status with a non-JSON body will pass the `!response.ok` check and be returned as `null` to the caller, which will then fail at the Zod parse step with a confusing error. Prefer re-throwing the parse error, or at minimum produce an `ApiError` that says "response body was not valid JSON" rather than a Zod validation error.

---

### 18. No workspace tooling at the repo root

There is no root `package.json`, `pnpm-workspace.yaml`, `turbo.json`, or equivalent. Each app must be installed and run independently. This is documented, but as Phases 5–9 add shared types (e.g., the diff change model), duplicating those between `apps/api` and `apps/web` will become painful. Consider adding a root workspace at some point.

---

### 19. Grammar nit in loading copy

**File:** `apps/web/src/pages/SchoolPage.tsx:122`

```tsx
"Backend renders both sources with Playwright, cleans page text, detect differs"
```

Should be: `"…cleans page text, and detects differences."` The typo is exposed to users during every diff load.

---

### 20. `POST` support is absent from the API client

**File:** `apps/web/src/services/api.ts`

Phase 8 requires `POST /api/schools/:schoolId/email-draft`. The `services/api.ts` module only exposes `getJson`. Add a `postJson` function now so it's ready and consistent with the existing client pattern, rather than adding ad-hoc `fetch` calls in feature query files later.

## Positive callouts

- **`StaffExtractionError`** with typed `status`, `code`, and `details` is a clean, typed exception pattern that makes the route handler easy to read.
- **Zod validation at both boundaries** (API env, scraper output, and frontend schema parse) is excellent — the codebase will fail loudly and clearly on contract violations.
- **`withStaffExtractionBrowser`** using a higher-order function with `try/finally` for browser lifecycle is correct and safe.
- **`phoneDedupKey`** normalizing the leading `1` country code before dedup shows careful attention to real-world data variance.
- **Test suite** using static HTML samples (not live sites) is the right call — stable, fast, and reproducible.
- **React Query cache settings** (`staleTime`, `gcTime`, `refetchOnWindowFocus: false`) are thoughtfully configured for the long-running diff query use case.
